# Лабораторный практикум: Сетевое программирование на C++ (SockWrapper)

## Общие требования ко всем задачам
- Каждое задание выполняется в отдельном каталоге
- Использовать реализованный на занятиях класс-обёртку `SocketWrapper` (`ServerSocket`, `ClientSocket`, `Socket`, `WSAInitializer`).
- Корректное закрытие сокетов во всех ветках исполнения (включая обработку исключений).
- Обязательная обработка сетевых ошибок (`NetworkException`, недоступность сервера, обрыв соединения).
- Возможность запуска клиента и сервера на `127.0.0.1` (localhost).
- Порт сервера вынести в константу или конфигурацию (по умолчанию `12345`).
- Код должен компилироваться под Windows с поддержкой C++17 или выше.
- Написать `ReadMe` к вашей работе с кратким описанием программы и инструкцией по сборке проекта

### Задача №1. Сервер точного времени (Time Sync)
**Описание:**  
Клиент запрашивает у сервера текущие дату и время. Сервер возвращает отформатированную строку. Задача демонстрирует базовый запрос-ответ и работу с системным временем.

**Протокол обмена:**  
- Клиент -> Сервер: `GET_TIME\n`  
- Сервер -> Клиент: `TIME: YYYY-MM-DD HH:MM:SS\n` или `ERROR\n`

**Базовый уровень:**  
- Сервер обрабатывает ровно одного клиента за сессию.
- После отправки времени сервер завершает соединение.
- Клиент выводит ответ в консоль и завершает работу.

**Асинхронный уровень:**  
- Сервер поддерживает пул потоков или создаёт отдельный поток для каждого подключённого клиента.
- Время на сервере обновляется в фоновом потоке раз в секунду и кешируется.
- При одновременном подключении N клиентов каждый получает ответ без задержек на обработку предыдущих запросов.

**Требования к логированию:**  
- Сервер: в `server.log` записывает `Timestamp | Client IP | Request: GET_TIME | Status: OK/ERROR`
- Клиент: в консоль выводит задержку между отправкой запроса и получением ответа (мс).

**Обработка ошибок:**  
- Некорректный запрос от клиента → сервер отвечает `ERROR\n` и закрывает соединение.
- Клиент отключается до получения ответа → сервер корректно закрывает сокет, не генерирует аварийного завершения.

### Задача №2. Удалённый калькулятор (Remote Calculator)
**Описание:**  
Клиент отправляет арифметическое выражение, сервер вычисляет результат и возвращает его. Задача направлена на парсинг сетевых пакетов и обработку бизнес-логики на стороне сервера.

**Протокол обмена:**  
- Клиент → Сервер: `CALC <a> <op> <b>\n` (пример: `CALC 10.5 * 2`)
- Сервер → Клиент: `RESULT: <value>\n` или `ERROR: <reason>\n`

**Базовый уровень:**  
- Поддерживаются операции: `+`, `-`, `*`, `/`.
- Сервер обрабатывает запросы последовательно в основном потоке.
- Деление на ноль возвращает `ERROR: division_by_zero`.

**Асинхронный уровень:**  
- Многопоточная обработка: каждый запрос выполняется в отдельном потоке.
- Введён искусственный таймаут вычисления: если операция занимает >3 сек (симулируется `std::this_thread::sleep_for`), сервер возвращает `ERROR: timeout`.
- Сервер ведёт потокобезопасный счётчик выполненных операций (`std::atomic<int>`), который выводится в консоль каждые 10 секунд.

**Требования к логированию:**  
- Сервер: `server_calc.log` содержит дату, IP, исходное выражение и результат.
- Клиент: при получении `ERROR` выводит причину в консоль с пометкой `[FAIL]`.

**Обработка ошибок:**  
- Нечисловые аргументы или неизвестный оператор → `ERROR: invalid_syntax`.
- Обрыв соединения во время вычисления → поток корректно завершается, сокет закрывается.

### Задача №3. Сервер хранилища заметок (Key-Value Store)
**Описание:**  
Простое in-memory хранилище типа `key:value`. Клиент может сохранять и запрашивать данные. Задача фокусируется на работе с разделяемыми данными в многопоточной среде.

**Протокол обмена:**  
- `SET <key> <value>\n` → `OK\n`
- `GET <key>\n` → `VALUE: <data>\n` или `NOT_FOUND\n`
- `DEL <key>\n` → `DELETED\n` или `NOT_FOUND\n`
- `QUIT\n` → сервер закрывает соединение с клиентом.

**Базовый уровень:**  
- Хранение в `std::map<std::string, std::string>`.
- Обработка одного клиента, последовательное выполнение команд.
- Данные теряются при перезапуске сервера.

**Асинхронный уровень:**  
- Потокобезопасный доступ к хранилищу через `std::shared_mutex` (reader-writer lock).
- Поддержка нескольких одновременных клиентов.
- Периодический сброс данных в файл `store_dump.txt` каждые 5 секунд или при получении сигнала `SHUTDOWN` (через отдельную команду или Ctrl+C).
- При старте сервер восстанавливает данные из файла, если он существует.

**Требования к логированию:**  
- Сервер: `server_kv.log` фиксирует все `SET/GET/DEL` с timestamp и IP.
- Клиент: выводит результат каждой команды. При `NOT_FOUND` помечает строку цветом/префиксом `[MISS]`.

**Обработка ошибок:**  
- Ключ содержит пробелы или пуст → `ERROR: invalid_key`.
- Попытка чтения пустого хранилища → `ERROR: storage_empty`.

### Задача №4. Сервер мониторинга процессов (Process Monitor)
**Описание:**  
Клиент отправляет имя исполняемого файла, сервер проверяет, запущен ли он в системе, и возвращает статус + базовую метрику (PID или % использования CPU).

**Протокол обмена:**  
- Клиент → Сервер: `STATUS <process_name>.exe\n`
- Сервер → Клиент: `RUNNING PID:<id> CPU:<percent>%\n` или `NOT_RUNNING\n`

**Базовый уровень:**  
- Проверка процесса через системные API (`CreateToolhelp32Snapshot` на Windows или `/proc` на Linux).
- Синхронная обработка одного запроса.
- Возвращает только статус `RUNNING`/`NOT_RUNNING`.

**Асинхронный уровень:**  
- Сервер запускает фоновый поток, который сканирует процессы раз в 2 секунды и сохраняет снимок в `std::unordered_map`.
- Клиентские запросы читают из кеша, не блокируя системные вызовы.
- Поддержка фильтрации: клиент может отправить `LIST\n` и получить CSV-список всех активных процессов с PID и CPU%.
- Многопоточность: до 10 одновременных соединений, обработка через `std::thread`.

**Требования к логированию:**  
- Сервер: `monitor.log` пишет каждый запрос, статус процесса и время обработки (мкс).
- Клиент: при получении `RUNNING` выводит форматированную таблицу: `| Process | PID | CPU% |`.

**Обработка ошибок:**  
- Неверный формат имени процесса → `ERROR: invalid_query`.
- Отсутствие прав доступа к чтению процессов → `ERROR: access_denied`.
- Клиент отключается → сервер не падает, логгирует `Client disconnected`.

### Задача №5. Широковещательный чат-хаб (Broadcast Chat)
**Описание:**  
Клиенты подключаются к серверу, отправляют текстовые сообщения. Сервер пересылает сообщение всем остальным подключённым клиентам. Задача демонстрирует работу с динамическим списком соединений и синхронизацию.

**Протокол обмена:**  
- При подключении клиент отправляет: `NICK <nickname>\n`
- Сервер отвечает: `WELCOME\n`
- Клиент отправляет текст: `<message>\n`
- Сервер рассылает всем: `[<nickname>] <message>\n`
- При отключении сервер рассылает: `[SYSTEM] <nickname> disconnected\n`

**Базовый уровень:**  
- Сервер работает в режиме эха: отправляет сообщение только отправителю.
- Поддерживает только одного клиента за сессию.
- Никнейм сохраняется в локальной переменной.

**Асинхронный уровень:**  
- Сервер хранит активные соединения в `std::vector<std::unique_ptr<Socket>>` или аналогичной структуре.
- Каждый клиент обслуживается в отдельном потоке.
- При получении сообщения от клиента сервер блокирует мьютекс списка, итерируется по всем сокетам и рассылает сообщение (кроме отправителя).
- Обработка "тихого" отключения: сервер периодически отправляет `PING\n`, при отсутствии ответа в течение 5 сек помечает клиента как offline и удаляет из списка.

**Требования к логированию:**  
- Сервер: `chat.log` сохраняет историю переписки в формате `[YYYY-MM-DD HH:MM:SS] [IP:Port] Nick: Message`.
- Клиент: выводит входящие сообщения с цветовой маркировкой или префиксом `>>`.

**Обработка ошибок:**  
- Дублирующийся никнейм -> `ERROR: nickname_taken`.
- Клиент отправляет пустую строку → сервер игнорирует, не рассылает.
- Сбой сети у одного клиента → сервер корректно закрывает его сокет, рассылает уведомление остальным, не прерывая работу хаба.
